# Spec: Retrieval + Intelligence Layer with Codebase Awareness

## Problem Statement

The current workflow (Plans 1–3) captures what was built and what was decided, but does not reduce the cost of understanding the codebase per ticket. Each planning phase re-reads relevant files from scratch. Discoveries and decisions accumulate in artifacts but are not surfaced at the point of need. For mature repositories this compounds into a significant per-ticket overhead: repeated codebase exploration, rediscovered constraints, and decisions made without access to prior reasoning.

The system needs a retrieval and intelligence layer that:
- Pre-computes code structure so Copilot does not re-read the repository per ticket
- Surfaces relevant prior knowledge (constraints, decisions, patterns) at the right workflow phase
- Scales from greenfield projects (minimal structure) to mature repositories (dense, interconnected systems)
- Remains inspectable, version-controlled, and maintainable without infrastructure

## Solution Approach

A hybrid model: a thin, engineer-triggered external script generates structured Markdown artifacts (indexes, module pages, knowledge pages, context packets) that Copilot reads during workflow phases. The script materialises structure and signals; Copilot does reasoning and synthesis.

Four layers:
1. **Generation Script** — parses source, extracts knowledge, assembles context packets
2. **Codebase Index** — pre-computed module structure and relationships
3. **Knowledge Index** — weighted historical signals from past artifacts
4. **Retrieval Protocol** — embedded in skills; governs how Copilot navigates the indexes

Everything is Markdown. Nothing runs continuously. No infrastructure beyond a command the engineer runs.

## Layer 1: Generation Script

### Command Interface

| Command | Jobs | Description |
|---|---|---|
| `index codebase` | A (full) | Full codebase parse and index regeneration |
| `index codebase --incremental` | A (incremental) | Re-parse only changed modules |
| `index knowledge` | B (full) | Full artifact scan and knowledge regeneration |
| `index knowledge --incremental` | B (incremental) | Process only new or changed artifacts |
| `index all` | A + B (full) | Full regeneration of both layers |
| `index all --incremental` | A + B (incremental) | Incremental update of both layers |
| `context-packet [ticket] [phase-N]` | C | Assemble context packet for specified phase |

### Job A — Codebase Parsing

Reads the source tree (path from `## Codebase → Entry points` in `conventions/SKILL.md`). Writes `codebase/index.md` and `codebase/[module].md` files.

**Module boundary detection:**
Auto-discovers boundaries by import structure (files imported together and never individually = one module). Before auto-discovery, checks `codebase/module-map.md` — any manually defined boundary takes precedence. Format: `[module-name] | [path(s)] | [responsibility]`.

**Responsibility inference (priority order):**
1. Module-level docstring or `README.md` in the module directory
2. `# Module:` or `# Responsibility:` comment at the top of the primary file
3. Inferred from public interface names
4. Fallback: `# inferred — verify this`

Inferred responsibilities are stored in `## Pending Validation` in the module page and shown as `responsibility: pending` in the index. They are not used by retrieval until an engineer promotes them to `## Responsibility`.

**Entry point detection:** controllers, handlers, scheduled jobs, event consumers, CLI entry points — detected by class naming convention and framework annotation scanning.

**Dependency type annotation:**

| Heuristic | Type |
|---|---|
| get, find, fetch, load, query | `(read)` |
| save, update, delete, create, publish, write | `(write)` |
| message queue, event bus, async annotation | `(async)` |
| default | `(sync)` |

Inferred types annotated `(type: X, confidence: inferred)`. Confirmed types: `(type: X)`.

**Metrics computed:**
- `Reach (direct)`: count of modules with a direct inbound edge
- `Downstream Reach (transitive)` per dependent: BFS count written to each Dependents entry
- Signal density placeholders (`0/0`) on first run — updated by Job B

**Source hash:** `hash(sorted ["path:mtime"] pairs for all source files)`. Stored in index header. Mismatch on next run → `stale: true`. Per-module hashes enable incremental mode: only regenerate pages for changed modules.

**Run triggers:** initial setup after `/setup`; after structural changes (new modules, deletions, large renames); on demand.

### Job B — Knowledge Extraction

Reads `artifact-index.md`, all plan `## Discoveries` and `## Amendments` sections, all spec `## Design Rationale` sections, all review BLOCKER flags. Writes `knowledge/index.md` and `knowledge/[topic].md`. Appends to `## Known Constraints` and `## Decisions` in module pages.

**Source routing:**

| Source | Knowledge type | Destination |
|---|---|---|
| `## Discoveries` | discovery | Knowledge index + module `## Known Constraints` |
| `## Amendments` | amendment | Knowledge index + module `## Known Constraints` or `## Decisions` (by content) |
| `## Design Rationale` (chosen) | decision | Module `## Decisions` only |
| `## Design Rationale` (rejected) | decision | Module `## Decisions` only ("Rejected:" prefix) |
| Review BLOCKER | blocker | Knowledge index + module `## Known Constraints` |
| Failed mitigation / rejected fix | discovery | Knowledge index + module `## Known Constraints`, tagged `[failed-mitigation]` or `[rejected-fix]` |

Amendment routing rule: if amendment changed an observed limit → `## Known Constraints`; if it changed a design choice → `## Decisions`.

**Module association:** exact class/module name match, then file path match via path-to-module map. No match → `module: unresolved`. Unresolved entries tracked in `knowledge/index.md` under `## Unresolved Entries`.

**Topic normalization:**
- Merge candidates: same module co-occurrence + overlapping keywords → flagged `merge-candidate`
- Split candidates: entries referencing modules with no dependency relationship + non-overlapping keywords → flagged `split-candidate`
- No auto-merging. Script flags; engineer confirms at review debrief with options: Merge, Split, Retain as-is

**Weight computation:**

```
recent = any entry with created-date within recent-window

weight:
  any entry impact = blocker                         → HIGH
  recurrence ≥ 3 AND recent = true                  → HIGH
  recurrence ≥ 3 AND recent = false                 → MEDIUM (historical)
  recurrence = 2                                    → MEDIUM
  recurrence = 1 AND impact = amendment             → MEDIUM
  otherwise                                         → LOW

instability: amendments on same topic ≥ 2           → status += "unstable"
pattern-candidate: recurrence ≥ 3, no pattern yet  → status += "pattern-candidate"
stale-candidate: weight = low AND no activity       → counted in stale-candidates header field
                 in last 10 tickets
```

**Signal density update:** after Job B completes, updates `Signals (recent/total)` column in `codebase/index.md` for each module.

**Incremental mode:** stores last-processed artifact content hash per artifact path. Only processes artifacts whose content hash has changed since the last run. Full rebuild required when topic normalizations are confirmed or module associations are corrected.

**Run triggers:** after closing each ticket (`index knowledge --incremental`); before starting a large ticket; full rebuild after major artifact restructuring.

### Job C — Context Packet Assembly

Reads the plan file for the specified ticket and phase, existing codebase and knowledge indexes. Writes `context/[ticket-id]/phase-[N]-context.md`.

**Trigger conditions (all must be true):**
- Repo maturity = medium or mature
- Plan annotated `Execution mode: phased`
- Phase has ≥ 4 files in its manifest

**Assembly:** maps each file in the phase manifest to a module via path-to-module map; loads module pages (condensed: Responsibility + Entry Points + Public Interface + Dependencies + Known Constraints + Decisions); loads high-weight knowledge entries for those modules (summary + pattern only, not full entry lists).

**Coverage confidence:**
- `high`: all phase files resolve to known modules; index age ≤ 7 days; no hash drift
- `medium`: one or more files unresolved; or index age 8–30 days
- `low`: majority unresolved; or index age > 30 days; or `stale: true` in index header

Every packet opens with: `_Note: module content is condensed. Full pages at codebase/[module].md. Load full pages if constraints or decisions require deeper context._`

## Layer 2: Codebase Index

### Tier 1 — `codebase/index.md`

**Header:**
```
<!-- generated: YYYY-MM-DD | modules: N | source-hash: [hash] | maturity: low|medium|mature | stale: true|false -->
```

**Maturity threshold:** `< 10 modules = low`, `10–30 = medium`, `> 30 = mature`. Override via `conventions/SKILL.md` field `Maturity threshold: N`. Default is fixed; override is explicit.

**Schema:**

| Module | Path | Responsibility | Reach (direct) | Signals (recent/total) | Quadrant | Risk Status |
|---|---|---|---|---|---|---|

- **Reach (direct):** inbound dependency count — direct imports only
- **Signals (recent/total):** `recent` = last 5 tickets or 90 days (whichever is broader)
- **Quadrant:** derived from Reach × total signals
  - High reach + high signals = critical-core
  - Low reach + high signals = sleeper-risk
  - High reach + low signals = stable-workhorse
  - Low reach + low signals = peripheral
- **Risk status:** `active` if recent > 0; `historical` if recent = 0 and total ≥ 1; `stable` otherwise

### Tier 2 — `codebase/[module].md`

**Sections (in order):**

```markdown
# [Module Name]
_Path: `[path]` | Generated: YYYY-MM-DD | Reach (direct): N | Quadrant: [quadrant] ([risk-status])_

## Responsibility
[What this module owns exclusively. What must not be implemented elsewhere.]

## Pending Validation
[Inferred responsibilities awaiting engineer confirmation. Empty once validated.]

## Entry Points
[Controllers, jobs, handlers that call into this module — with the specific method called.]

## Public Interface
[Key classes, methods, endpoints exposed to other modules.]

## Dependencies
- `[Module]` (type: [read|write|async|sync][, confidence: inferred]) — [relationship annotation]

## Dependents
- `[Module]` — [why it depends on this] | Downstream Reach (transitive): N

## Known Constraints
_(Observed limitations — non-negotiable. Override requires architectural discussion.)_
- [YYYY-MM-DD] [TICKET]: [constraint statement]

## Decisions
_(Chosen designs — override allowed with explicit justification recorded in spec.)_
- [YYYY-MM-DD] [TICKET]: [decision statement]
  Rejected: [alternative] — [reason]
  Source: `[spec-path]`
  Status: superseded (see [TICKET-A], [TICKET-B])  ← added when override chain ≥ 2
```

### Relationship Model

Relationships are bidirectional, encoded within module pages. The distributed model is navigable via the index without loading a global graph.

| Question | Navigation |
|---|---|
| What depends on X? | Read X's `## Dependents` |
| What does X need? | Read X's `## Dependencies` |
| What is impacted by changing X? | Read X's `## Dependents`, note Downstream Reach (transitive) figures |
| Why was X built this way? | Read X's `## Decisions` |
| Which modules are highest-risk? | Sort codebase index by Reach (direct), filter to active risk status |

## Layer 3: Knowledge Index

### Tier 1 — `knowledge/index.md`

**Header:**
```
<!-- generated: YYYY-MM-DD | topics: N | recent-window: last-5-tickets OR 90-days | pattern-candidates: N | stale-candidates: N | unresolved: N -->
```

**Schema:**

| Topic | Module(s) | Weight | Recurrence | Recent | Impact | Last Ticket | Status |
|---|---|---|---|---|---|---|---|

Status values: `active`, `historical`, `unstable`, `pattern-candidate` (combinable, e.g., `active, pattern-candidate`).

**Unresolved entries:** listed in `## Unresolved Entries` section at the bottom of the file. Does not affect indexed topics.

### Tier 2 — `knowledge/[topic].md`

```markdown
# [Topic Name]
_(Topic represents a single root cause or invariant failure mode — not a symptom.)_
_Modules: [list] | Weight: [high|medium|low] | Recurrence: N | Status: [status]_

## Summary
[1–2 sentence synthesis of the root cause and its observable effect.]

## Pattern
_(Present only when status includes pattern-candidate and synthesis is confirmed.)_
_Confidence: [high|medium|low]_
[One-sentence pattern statement.]

[Confidence rubric:
  High   = same root cause, same module(s), same resolution pattern, ≥3 tickets
  Medium = same symptom, different resolutions, partial consistency
  Low    = weak correlation, early signal, not yet reliable]

## Entries
_(High-weight first, then chronological within weight tier.)_
- [YYYY-MM-DD] [TICKET] | [type] | [Weight] | [description]
  [failed-mitigation] or [rejected-fix] tag where applicable

## Related Modules
- [`[Module]`](../codebase/[module].md) — [relationship to this topic]
```

### Topic Normalization Rules

A topic represents one root cause or invariant failure mode. Merge when entries share the same root cause across different tickets despite different surface symptoms. Split when a topic name encompasses two independent failure modes that never co-occur. "Retain as-is" is always a valid explicit outcome at review debrief, recorded as `normalization: reviewed-no-change`.

## Layer 4: Retrieval Protocol

### Activation Gate

```
1. Does codebase/index.md exist?
   No → skip codebase protocol. Fall back to codebase search protocol (Plan 3).
2. Read index header. What is maturity?
   low → skip codebase protocol.
   medium or mature → run full protocol.
```

### Loading Budget (hard limits)

| Phase | Module pages | Knowledge pages | Expansion |
|---|---|---|---|
| Brainstorming | 0 (index-only) | 0 (index-only) | None |
| Planning | 3 | 2 | +1 module, +1 knowledge |
| Spec-writing | 0 (## Decisions only) | 0 | None |
| Execution | Via context packet | Via context packet | None |

### Ticket Type Classification (deterministic)

Applied to brainstorm artifact `## Problem` section. First match wins:
1. Contains "fix", "bug", "error", "fail" → `bug-fix`
2. Contains "modify", "refactor", "update" → `modification`
3. Otherwise → `new-feature`

Bias on loading split:
- `new-feature` → 3 module pages, 1 knowledge page
- `modification` → 2 module pages, 2 knowledge pages
- `bug-fix` → 1 module page, 2 knowledge pages

### Navigation Decision Tree (Planning Phase)

```
1. READ brainstorm artifact
   → Extract named modules, file paths, feature area keywords
   → Classify ticket type

2. READ codebase/index.md
   → Identify candidate modules:
     a. Directly named or path-matched in brainstorm
     b. Active-risk modules whose Responsibility overlaps feature area
     c. Critical-core modules with the highest Downstream Reach (transitive) adjacent to the feature area
   → If candidate pool empty: select one fallback — highest Reach (direct),
     broken by closest Responsibility keyword match to feature area

3. APPLY ticket-type bias to set module/knowledge loading split

4. LOAD module pages in priority order (within budget):
   Priority 1: directly named modules
   Priority 2: active-risk modules in the feature area
   Priority 3: critical-core modules adjacent to named modules

5. READ knowledge/index.md
   → Filter to loaded modules
   → Sort: high-weight first, active before historical, recent before stale
   → Exclude low-weight entries unless no medium/high entries exist for loaded modules
   → If no entries exist for loaded modules: skip knowledge loading;
     note in retrieval summary: "Knowledge loaded: none"

6. LOAD knowledge pages within budget

7. RUN expansion check (see below)

8. PRODUCE retrieval summary

9. STOP. Uncertainty remaining is resolved during execution, not by
   expanding retrieval beyond the defined budget.
```

### One-Hop Expansion

**Mismatch triggers (any one activates expansion):**
- Loaded module page has no class or interface relevant to the feature area
- Knowledge index has a high-weight entry for an unloaded adjacent module matching the feature area
- A loaded module's `## Dependencies` or `## Dependents` references an unloaded critical-core or sleeper-risk module
- Loaded signals describe a different problem domain than the brainstorm feature area (negative signal)

**Candidate selection (strict priority — first decisive criterion wins):**
1. Build pool: all unloaded modules within one dependency hop of the loaded set
2. Filter to critical-core or sleeper-risk quadrants — if none, skip expansion entirely
3. Prefer active-risk over historical
4. Prefer outbound edges over inbound-only (the feature is built into downstream modules)
5. Prefer candidates with high-weight knowledge entries matching the feature area
6. Select top 1

Load the selected module's tier-2 page. If a high-weight knowledge entry exists for it matching the feature area, load that knowledge page too. Maximum one expansion. No second hop.

### Retrieval Summary Format

```markdown
## Intelligence Context
- Index maturity: [low|medium|mature] ([N] modules)
- Index age: [N] days | Hash drift: [none|detected]
- Ticket type: [new-feature|modification|bug-fix]
- Modules loaded: [list]
- Knowledge loaded: [list with weights] | or "none — no entries indexed for loaded modules"
- Expansion: [none] | [module name] — [reason: quadrant + trigger]
- Coverage confidence: [high|medium|low]
- Warnings: [staleness flags, missing modules, pending validation markers]
```

### Staleness Handling

| Signal | Trigger | Behavior |
|---|---|---|
| Missing module | Module named in brainstorm not in index | Log warning; fall back to codebase search protocol for that module only |
| Hash drift | Index header shows `stale: true` | Prepend warning to retrieval summary |
| Age warning | Index age > 30 days | Context packet prepends age warning |
| Index absent | File does not exist | Skip protocol; direct exploration only |

## Workflow Integration

### New Conventions Fields

Add to `conventions/SKILL.md` Artifact Paths:
```
Codebase Index:  <e.g. codebase/>
Knowledge Index: <e.g. knowledge/>
Context Packets: <e.g. context/>
```

Setup skill gains detection logic for these three paths; defaults to `codebase/`, `knowledge/`, `context/` if absent.

### Per-Skill Integration

**Brainstorming skill** — index-only scan before first question:
```
1. Read codebase/index.md. If absent or maturity = low: skip to step 3.
2. Identify candidate modules. Surface quadrant, risk status, high-weight signals.
3. Read knowledge/index.md for candidate modules. Surface high-weight entries only.
4. Open with suggestive framing:
   "Based on the index, [module X] appears to be the primary area for this work.
   It is flagged as [quadrant] with [N] recent signals.
   Known signals: [one-line summaries]. Does this match your understanding?"
   If no candidates found: open without a codebase statement.
```

**Spec-writing skill** — decision conflict check after architecture is drafted:
```
1. Identify modules in scope from the Architecture section.
2. Read ## Decisions from each relevant module page.
3. Check for conflicts: direct contradiction, weakening of assumptions, or
   re-proposing an explicitly rejected alternative.
4. If conflict:
   Flag with specific decision text and options A (revise) or B (override with reason).
   Do not finalize spec until engineer selects.
5. If no conflict: note "Decision check: no conflicts found."
Override path B: add ## Design Override to spec; picked up by Job B on next run.
```

**Planning skill** — full retrieval protocol before plan is drafted:
```
1. Run full retrieval protocol. Append retrieval summary as ## Intelligence Context
   in the plan preamble.
2. Known Constraints from loaded module pages → plan ## Constraints section,
   labelled "[source: module-page]". Non-negotiable.
3. High-weight knowledge entries → inform phase ordering and risk notes.
   Knowledge informs constraints and risks; it does not dictate plan structure.
4. Run decision conflict check before finalizing (same as spec-writing).
```

**Execution skill** — context packet read at phase start:
```
1. Check for context/[ticket-id]/phase-[N]-context.md.
   If found: read it; note coverage confidence; proceed with its content.
   If not found: no pre-loaded context; codebase search protocol (Plan 3) active on demand.
2. Before any change affecting a public interface, dependency behavior,
   or constraint boundary: confirm alignment with ## Relevant Decisions in the packet.
   If deviation: stop and surface conflict; do not proceed without acknowledgment.
```

**Review skill** — intelligence maintenance added to post-ticket debrief:
```
5. Check knowledge/index.md for pattern-candidates tagged to modules touched this ticket.
   If any: surface entries; prompt synthesis with options:
   A — Synthesize (write ## Pattern with confidence level)
   B — Retain as-is (mark normalization: reviewed-no-change)
   C — Merge with [topic]
   D — Split into [topics]
   E — Defer (insufficient signal; revisit after N additional tickets)
6. After debrief: run index knowledge --incremental
```

## Requirements

Each requirement is testable by tracing through the design.

1. **R1:** `codebase/index.md` always contains a maturity header, source-hash, and generation date. A skill reading the header can determine whether to run the retrieval protocol without reading any other file.

2. **R2:** Every module page answers four questions without loading any other file: what exists, what happened, why it was built this way, and what depends on it.

3. **R3:** The retrieval protocol always terminates within the defined loading budget. Under no path does it load more than (budget + 1 expansion) files.

4. **R4:** Every knowledge entry has an explicit weight derived deterministically from recurrence, impact, and recency. No entry is weight-ambiguous.

5. **R5:** A decision conflict is always surfaced before a spec or plan is finalized, when a relevant `## Decisions` entry exists for a module in scope.

6. **R6:** Context packets include a coverage confidence field and an explicit disclosure that content is condensed, with a reference to full module pages.

7. **R7:** The knowledge index header always surfaces `unresolved: N` and `stale-candidates: N`. These fields are never absent.

8. **R8:** The retrieval summary is always appended to planning phase output. It is deterministic: the same brainstorm artifact + same indexes produce the same summary.

9. **R9:** Cross-repo knowledge sharing is not supported. Each repository is an independent knowledge boundary. This is a permanent design constraint, not a deferral.

10. **R10:** The generation script never modifies business logic or workflow artifact content. It reads source files and existing artifacts; it only writes to the index layer.

## Architecture / Design Decisions

### Hybrid model over pure Copilot

The external script generates Markdown artifacts that Copilot reads. This sidesteps Copilot's inability to query or cache codebase structure natively (C3, C4 in the gap analysis). The alternative — encoding all structure as skill instructions — would be unmaintainable at scale.

### Distributed graph over central graph file

Relationships are encoded bidirectionally in module pages rather than in a separate graph file. A central graph file grows without bound and would need to be loaded in full to answer any single question. The distributed model lets Copilot load only the pages relevant to the current task.

### Human-in-the-loop for pattern synthesis

The script identifies pattern candidates; engineers synthesize patterns. Automated synthesis of recurring signals into pattern statements risks false consolidation — entries that share a symptom but have different root causes. Human synthesis is slower but more accurate. The debrief trigger ensures synthesis happens at maximum context.

### Decisions as constraints by default

Recorded architectural decisions are treated as active constraints requiring explicit override, not passive information. The alternative — treating all decisions as advisory — would allow the system to drift silently from its own architectural commitments without any signal. The override path always exists; the check ensures drift is deliberate.

## Design Rationale

- **Chosen: two-tier index (top-level + per-module/topic pages)** over monolithic snapshot files — monolithic files grow without bound and load everything even when only a fraction is relevant. Two-tier enables scoped loading.
- **Rejected: monolithic snapshot** — file size degrades with repo maturity; entire file loaded even for one-module questions.
- **Chosen: distributed relationship model** over separate graph file — navigable without global load; scales with module count.
- **Rejected: single graph file** — grows without bound; must be fully loaded to answer any traversal question.
- **Chosen: human pattern synthesis** over automated — prevents false pattern consolidation from symptom-only matching.
- **Rejected: automated pattern synthesis** — recurrence without root cause alignment produces misleading patterns.
- **Chosen: hard loading budget** over soft guidance — bounded loading is reliable; soft guidance is overridden under pressure.
- **Rejected: soft guidance** — agents tend to load more when uncertain; budget creep recreates the full-repo-read problem.

## Risks & Dependencies

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Module boundary misalignment on first setup | High | Medium | `module-map.md` override; `(inferred)` markers on pending validation |
| Stale indexes serving wrong context | Medium | Medium | Hash drift detection; age warnings; coverage confidence |
| Pattern false positives | Low | Medium | Human synthesis step; Defer option at debrief |
| Decision over-accumulation causing friction | Medium | Low | Override path always available; engineer is never blocked |
| Knowledge noise from low-weight entries | Medium | Low | Weight capping; stale-candidates field signals pruning need |
| Context packet staleness | Medium | Medium | Coverage confidence field; staleness warning in packet |
| Engineers skipping debrief (no pattern synthesis) | High | Medium | Visible in index header (pattern-candidates: N); no automated mitigation |

**Dependencies:**
- Plans 1–3 must be implemented first. Job B depends on `artifact-index.md`, `## Discoveries`, `## Amendments`, and `## Design Rationale` sections that Plans 1–3 add.
- `conventions/SKILL.md` must include the three new artifact paths before skills can use them.
- The generation script depends on the source tree being accessible at the path defined in `conventions/SKILL.md → ## Codebase → Entry points`.

## Testing Strategy

This system is tested by scenario trace (same method as Plans 1–3 — no code, no build system).

**Scenario 1 — Full lifecycle on a mature repo:**
Run `index all` on a repo with 15 modules and 10 completed tickets. Verify: `codebase/index.md` has 15 rows; each module page has `## Dependents` with Downstream Reach figures; `knowledge/index.md` has entries with correct weights; at least one `pattern-candidate` if recurrence ≥ 3 exists.

**Scenario 2 — Retrieval protocol for a bug-fix ticket:**
Brainstorm artifact Problem section: "fix token validation failure." Ticket type classifies as `bug-fix`. Verify: 1 module page loaded (AuthService), 2 knowledge pages loaded (high-weight entries for AuthService), expansion check runs, retrieval summary appended to plan with `Coverage confidence`.

**Scenario 3 — Decision conflict detection:**
Spec proposes in-memory cache for AuthService. `codebase/auth-service.md` has a recorded `## Decisions` entry rejecting in-memory cache. Verify: conflict is flagged before spec is finalized; engineer selects override or revises; if override, `## Design Override` section appears in spec.

**Scenario 4 — Context packet for execution:**
Ticket PROJ-123, Phase 2, 5 files across 3 modules. `context-packet PROJ-123 phase-2` runs. Verify: packet contains condensed module pages for all 3 modules; high-weight knowledge entries included; coverage confidence computed correctly; packet opens with condensed-content disclosure.

**Scenario 5 — Greenfield behavior:**
Repo with 6 modules (maturity = low). Verify: retrieval protocol skips codebase layer in brainstorm and planning; context packets are not generated; system falls through to codebase search protocol (Plan 3) on demand.

**Scenario 6 — Pattern synthesis at review:**
Topic `token-invalidation-lag` reaches recurrence = 3. Review debrief surfaces it as `pattern-candidate`. Engineer selects Synthesize, writes `## Pattern` section with `confidence: high`. Job B incremental run on next `index knowledge --incremental` picks up the change. Verify: knowledge index status for that topic updates from `pattern-candidate` to `active`.

**Scenario 7 — Stale index detection:**
Source file modified after last `index codebase` run. Next incremental run detects hash mismatch. Verify: `stale: true` appears in index header; retrieval summary for next planning phase includes staleness warning.

## Open Questions (Resolved)

All five open questions from the design session have been resolved and incorporated into the design:

1. **Pattern confidence rubric** — defined in `## Pattern` section schema and this spec's Requirements.
2. **Decision supersession** — `Status: superseded (see TICKET)` added when override chain ≥ 2.
3. **Knowledge pruning** — `stale-candidates: N` surfaces the signal; pruning policy deferred to post-implementation with real data.
4. **Maturity threshold** — fixed default with explicit override only via `conventions/SKILL.md`.
5. **Cross-repo knowledge** — permanently out of scope; R9 encodes this as a design constraint.
